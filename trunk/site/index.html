Guttershark - Actionscript 3 Library

WHAT THE FUCK?

I know what you're thinking. Another Library? Hear me out - Guttershark is a library that pushes convention to make you faster as a developer. Numerous other AS3 libraries are out there but they all try to push insane amounts of abstraction on you.

Just because something can be abstracted, doesn't mean it should be. All these libraries implement about every design pattern you can find, just because they can. Does that make it more useful? Probably not.

If something that is so abstracted is supposed to so useful, why does it take so long just to implement some really basic functionality?

So what's the deal with Guttershark? Everything in the Guttershark library is abstract enough that you can piece something together with the libraries included. But the real benefit comes from using the conventions defined by it. The biggest benefit is the amount of time it takes to get up and running with a complete site.

By following the conventions, you can override literally one method, write about 6 lines of code, and have the following implemented:

  • Loads a "site XML" file.
  • Site XML parsers for simplification and convention of data extraction.
  • Preloading Framework.
  • Bandwidth Detection (High/Med/Low).
  • CPU Speed Estimation (Fast/Med/Slow).
  • Online Status Pinging.
  • Akamai Ident Service Integration.
  • Akamai Net Connection Manager for FLVPlayback components.
  • Simplified Keyboard Event Management.
  • Simplified Language Management.

All of the above options are determined by an opt-in system through FlashVars.

So, by conforming to some pretty loose conventions, you get a lot of things for free, and still have plenty of room to implement anything else you wish.

DOCUMENTATION PHILOSOPHY

My philosophy behind documentation is that it should make sense, and allow me to tell someone to look at the API docs, and not have to answer any questions.

How can that happen? By having an extremely well documented API. 95% of classes have examples, and explain things in a readable fashion. The guide for writing documentation for Guttershark has been the Flash API. What can makes more sense than following how Adobe does it?

Documentation is always updated and maintained.
